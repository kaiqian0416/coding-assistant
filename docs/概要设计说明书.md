### 1.3 设计原则

为规避往届开发常见缺陷，本项目遵循以下设计原则：

| 缺陷 | 本项目的应对措施 |
|------|-----------------|
| **代码书写风格混乱** | 统一使用Lombok简化POJO、统一命名规范（驼峰+下划线自动映射）、统一CSS变量管理主题色、统一笔名为所有Java类添加Javadoc |
| **模块耦合严重** | 严格Controller→Service→Mapper分层，Service面向接口编程（接口+实现类分离），前端Pinia独立状态管理，组件间通过Props/API通信 |
| **测试覆盖不足** | 全局异常处理器、参数校验注解（JSR-303）、API响应统一封装（ApiResponse）、前后端双重权限校验 |

### 3.2 模块依赖关系

```
Controller ──→ Service接口 ←── Service实现 ──→ Mapper ──→ Entity/DTO
                 ↕  (依赖注入，单例，构造器注入)
              Config（跨域/拦截器/安全）
```

### 3.3 解耦设计亮点

1. **面向接口编程**：Service层全部定义接口+实现类分离，Controller依赖Service接口
2. **依赖注入**：统一使用构造器注入（Spring推荐最佳实践）
3. **AI服务抽象**：AiService定义统一接口，实现类内部处理真实API/模拟回退，外部无感知
4. **配置外部化**：API Key、URL通过application.properties配置，不硬编码
5. **统一响应格式**：ApiResponse<T> 封装所有HTTP响应，前端统一处理
6. **CSS变量双层主题**：`:root` / `[data-theme="dark"]` / `[data-theme="light"]` 三套变量，单行class切换全局主题

## 4. 数据库设计

### 4.1 E-R图

```
user (1) ──→ (N) submission      ← 一个用户多次提交
question (1) ──→ (N) submission  ← 一道题被多次提交
user (1) ──→ (N) learning_stats  ← 一个用户有多个知识点统计
achievement (独立配置表)          ← 成就定义，系统自动计算用户是否满足
```

### 4.2 表结构清单

| 表名 | 说明 | 关键字段 | 索引 |
|------|------|---------|------|
| **user** | 用户表 | id(PK), username(UK), password(BCrypt), role(user/admin), ability_level | uk_username, uk_email |
| **question** | 题目表 | id(PK), title, difficulty, knowledge_point, review_status | idx_difficulty, idx_knowledge_point, idx_review_status |
| **submission** | 提交记录表 | id(PK), user_id(FK), question_id(FK), code, result, score, ai_diagnosis | idx_user_id, idx_question_id, idx_user_question |
| **learning_stats** | 学习统计表 | id(PK), user_id(FK), knowledge_point, total/correct/wrong_count, accuracy | uk_user_knowledge, idx_user_id |
| **achievement** | 成就定义表 | id(PK), name, icon, condition_type, condition_value, sort_order | idx_condition |

### 4.3 数据库设计要点

- **密码安全**：BCrypt(10)加密，不可逆
- **唯一约束**：username、email设UNIQUE，空email存null（MySQL允许多个null）
- **审核流程**：question表review_status字段（pending→approved/rejected），保证AI生成题目需人工审核
- **统计更新**：每次提交后自动更新learning_stats表（INSERT OR UPDATE）
- **可扩展成就**：achievement表支持管理员动态配置，condition_type支持多种统计维度

## 6. 关键业务流程

### 6.1 代码提交与判题流程（核心）

```
POST /api/submissions {questionId, code, language:'python'}
    ↓
① 校验管理员角色 → 禁止提交
② 获取题目信息（question）
③ 创建提交记录（result='judging', score=0）
④ 判题核心（SubmissionServiceImpl.judge）:
    ├── 真实Python执行:
    │   └── 写入临时.py文件 → python3执行 → 传入样例输入 → 捕获stdout
    └── 回退模拟判题:
        └── 分析print()语句 → 提取输出文本
⑤ 与预期输出对比 → 计算得分
⑥ AI代码审查（AiService.reviewCode）:
    ├── 真实API: 发送代码给AI判断
    └── 模拟模式: 检查input()/变量/print()模式
⑦ 硬编码→降分到最高15分
⑧ 组装诊断+审查反馈（aiDiagnosis字段）
⑨ 更新learning_stats表
⑩ 返回完整JSON结果
```

### 6.2 个性化推荐流程

```
GET /api/questions/recommended?userId={id}
    ↓
① 获取用户ability_level
② 从learning_stats获取用户薄弱知识点（TOP3，按正确率升序）
③ 从question表按知识点+难度匹配查询（ORDER BY RAND()）
④ 不足5题时补充随机题目
⑤ 返回去重后的5道题目
```

### 6.3 AI辅助出题流程（管理员）

```
POST /api/admin/ai/generate-question {knowledgePoint, difficulty, topic}
    ↓
① 构造Prompt发送到智谱AI
② AI返回JSON（题目描述/样例/参考代码）
③ 解析JSON（自动清理```json标记）
④ 存入question表（source='ai_generated', review_status='pending'）
⑤ 管理员预览生成结果
⑥ 管理员操作:
    ├── 审核通过 → review_status='approved' → 对用户可见
    ├── 手动修改 → 编辑后保存
    └── 驳回 → review_status='rejected'
```

## 8. 安全设计

### 8.1 认证与授权

| 层级 | 机制 |
|------|------|
| 密码存储 | BCrypt(10)加密，不可逆 |
| 登录验证 | 用户名+BCrypt比对 |
| 会话管理 | HttpSession，login写入/logout销毁 |
| 接口拦截 | SessionInterceptor检查Session中loginUser |
| 管理员校验 | Controller层requireAdmin() + User角色字段双重校验 |
| 不能删除自己 | AdminController中校验 admin.getId().equals(id) |

### 8.2 数据校验

| 层级 | 机制 |
|------|------|
| 前端校验 | 表单required、pattern、v-model绑定 |
| 后端校验 | JSR-303注解（@NotBlank、@Size、@Valid） |
| 全局异常处理 | GlobalExceptionHandler捕获MethodArgumentNotValidException返回中文提示 |
| 业务校验 | Service层检查用户名/邮箱重复、原密码正确性 |

### 8.3 防攻击设计

- 密码BCrypt加盐，防彩虹表
- 管理员禁做提交，防刷分
- 空邮箱存null，防UNIQUE约束SQL注入
- CORS限制`/api/**`路径，防跨站
- Python代码在临时文件中执行，进程隔离
